Deploying Agentic AI systems and scaling LLMs exposes a brutal reality of MLOps: compute is rarely your actual bottleneck—memory is. Specifically, VRAM capacity dictates how many concurrent users or agents you can serve.

In autoregressive generation, we cache the representations of past tokens so we don’t have to recompute them. This is the KV Cache. Historically, AI frameworks treated the KV Cache like any standard tensor: it had to live in a single, contiguous block of VRAM. This strict requirement created massive memory fragmentation and bottlenecked throughput.

PagedAttention is the breakthrough that solved this by borrowing a classic concept from operating systems: virtual memory paging.

In this post, I will break down the limitations of the traditional KV cache, explain how PagedAttention fixes it using Block Tables, walk through a real-world analogy, follow one toy generation request step by step, and share a simplified PyTorch mental model of the forward pass.

1. The Problem: Traditional KV Cache Allocation

When an LLM generates text, it reads the prompt and outputs one token at a time. To avoid recalculating the Key (K) and Value (V) vectors for every previous token at every single step, we store them in the KV Cache.

The Contiguous Memory Rule

Before PagedAttention (in vanilla PyTorch or HuggingFace inference), the KV cache tensor was subjected to standard matrix physics: tensors require contiguous memory.

Because sequence lengths are highly unpredictable (you never know if an agent will generate a “Yes” or a 2,000-word essay), the inference engine had a terrible choice to make:

  • Guess small: Risk running out of memory (OOM) mid-generation, crashing the request.
  • Over-reserve: Pre-allocate a massive, contiguous chunk of VRAM equal to the model’s maximum context length for every single request.

The Memory Wall (Examples of Waste)

Pre-allocation causes catastrophic memory waste through two mechanisms:

  • Internal Fragmentation: You pre-allocate space for 2,048 tokens. The user asks a simple question, and the LLM responds with 18 tokens. The remaining 2,030 token slots are locked, sitting empty, and completely unusable by other requests.
  • External Fragmentation: Over time, VRAM becomes a checkerboard of used and empty spaces. You might have 5GB of total free VRAM, but because it is broken up into tiny gaps between active tensors, you cannot find a single 2GB contiguous block to start a new request.

The Result: Studies showed that in standard inference servers, up to 80% of the KV Cache VRAM was wasted.

2. The Solution: PagedAttention

PagedAttention removes the requirement for contiguous memory. It breaks the KV cache into small, fixed-size chunks called blocks (typically holding 16 or 32 tokens).

A Real-Life Analogy: The Restaurant Seating Problem

  • The Traditional KV Cache: Imagine a restaurant that only has massive, 20-person banquet tables. If a couple walks in for dinner, the host seats them at a 20-person table. 18 chairs are completely wasted. If another couple walks in, they cannot sit at the empty chairs of the first table; they need their own giant 20-person table. The restaurant is “full” after seating just a few people.
  • PagedAttention: The restaurant uses small, 4-person tables that can be logically grouped. When a couple walks in, they get one 4-person table. If a party of 10 walks in, the host gives them three 4-person tables scattered wherever there is space across the room. The waiter (the Block Table) simply remembers which scattered tables belong to which party and serves them in order. The restaurant can now operate at near 100% capacity.

How it Works Under the Hood

  1. Logical vs. Physical: To the Transformer’s math operations, the sequence still looks like one long, continuous (logical) sequence. In reality, it is scattered across non-contiguous (physical) blocks in VRAM.
  2. The Block Table: PagedAttention maintains a translation map called the Block Table. It records which physical VRAM blocks belong to which logical sequence.
  3. Dynamic Allocation: When an agent starts generating text, the engine allocates just one small physical block. Once that block fills up with tokens, it allocates a new one anywhere in VRAM and updates the Block Table.

3. Walkthrough: One Request, Step by Step

The restaurant analogy explains the idea; a concrete token run shows how the Block Table and physical blocks move in lockstep. Production systems often use blocks of 16 or 32 tokens. For readability, this walkthrough assumes block size 4—the bookkeeping is identical, just with smaller numbers.

Suppose one user sends a short prompt and the model will decode two new tokens afterward.

  • Prompt (prefill): ["The", " quick", " brown", " fox"]
  • Target generation (decode): [" jumps", " over"]

Step 1: Prefill—processing the prompt

Before anything new is generated, the engine must ingest the prompt.

Computation. The Transformer processes all four prompt tokens together (this is the prefill phase). For each token it computes the Query (Q), Key (K), and Value (V) vectors.

Allocation. The scheduler looks at its pool of physical VRAM blocks and picks an empty one—say physical block 12.

Storage. It writes the K and V entries for all four prompt tokens into that block. With block size 4, the first logical block is now full.

Block table. For this request, the engine records:

  • Logical block 0 \(\rightarrow\) physical block 12 (slots used: 4/4)

So far, this looks like ordinary caching—except the “row” in VRAM is exactly one small block, not a giant pre-allocated tensor for max context length.

Step 2: First decode token—crossing a block boundary

Now the model enters the autoregressive loop and produces the first new token.

Generation. Given the prompt context, the language head predicts the next token: " jumps".

Computation. The model computes K and V for " jumps". Those vectors must be stored in the KV cache for every later step.

Why allocate again? The block table says logical block 0 maps to physical block 12, and that block is already 4/4. There is no room for another token in that physical page, so the engine must grab a new physical block. Suppose free space shows up at physical block 3.

Storage. The K and V tensors for " jumps" go into the first slot of physical block 3.

Block table after this step:

  • Logical block 0 \(\rightarrow\) physical block 12 (slots used: 4/4)
  • Logical block 1 \(\rightarrow\) physical block 3 (slots used: 1/4)

Notice the sequence is still logically one continuous string of five tokens; physically, the first four live in block 12 and the fifth lives at the start of block 3.

Step 3: Second decode token—where PagedAttention shows up in the math

The model now continues from the full context ["The", " quick", " brown", " fox", " jumps"] and must emit the next token.

Query. From the current position (the token " jumps"), the model computes the Q vector for the next attention step.

Attention. To form attention scores, that Q must interact with the K vectors of all prior tokens—four from the prompt plus " jumps".

Fetch (the important part). A naive layout would assume one contiguous K tensor in VRAM. PagedAttention does not. The custom kernel consults the block table and issues the reads it actually needs—for example:

  1. Go to physical block 12 and read four K/V slots (the prompt).
  2. Jump to physical block 3 and read one K/V slot (" jumps").

The softmax and the weighted sum over values still run on the same mathematical sequence; only the memory access pattern is paged.

Generation. After attention and the feed-forward layers, the model predicts " over".

Storage. K and V for " over" are computed. Logical block 1 still has three free slots in physical block 3, so no new physical block is required—the new vectors simply occupy the second slot in block 3.

Block table after this step:

  • Logical block 0 \(\rightarrow\) physical block 12 (slots used: 4/4)
  • Logical block 1 \(\rightarrow\) physical block 3 (slots used: 2/4)

The system-level view

While this one request was using physical blocks 12 and 3, other users’ sequences are decoding in parallel, filling blocks 1, 2, 4, 13, and so on. The allocator hands out small pages wherever they fit; the block table per request is what preserves the illusion of a single contiguous KV tensor for attention math.

Treating the KV cache like virtual memory—logical order mapped to scattered physical pages—is how the engine packs VRAM tightly without pre-allocating one enormous contiguous tensor per request or repeatedly forcing expensive torch.cat-style reallocations when sequences grow.

4. A Conceptual PyTorch Implementation

You cannot use a standard torch.matmul(Q, K.T) on scattered memory. PagedAttention requires custom CUDA/Triton kernels to read the Block Table and fetch the scattered chunks during the attention computation.

Here is a simplified PyTorch pseudo-code block to illustrate how this logic looks inside the inference loop.

import torch
import torch.nn.functional as F

def paged_attention_forward(
    query: torch.Tensor,               # (num_seqs, num_heads, head_size)
    key_cache: torch.Tensor,           # (num_blocks, block_size, num_heads, head_size)
    value_cache: torch.Tensor,         # (num_blocks, block_size, num_heads, head_size)
    block_tables: torch.Tensor,        # (num_seqs, max_blocks_per_seq) -> Maps sequence to physical blocks
    context_lens: torch.Tensor,        # (num_seqs) -> How many tokens in each sequence
    block_size: int
) -> torch.Tensor:
    
    num_seqs, num_heads, head_size = query.shape
    out = torch.zeros_like(query)

    # In reality, this loop is aggressively parallelized in a custom CUDA kernel
    for i in range(num_seqs):
        seq_len = context_lens[i].item()
        
        # 1. Fetch exactly which physical blocks this sequence is scattered across
        num_blocks_needed = (seq_len + block_size - 1) // block_size
        physical_block_indices = block_tables[i, :num_blocks_needed]
        
        # 2. Gather the scattered Key and Value chunks into temporary contiguous tensors
        # (A real CUDA kernel does the dot product directly on the fly without this copy step!)
        K_seq = key_cache[physical_block_indices]    # (blocks, block_size, heads, dim)
        V_seq = value_cache[physical_block_indices]
        
        # Flatten blocks to look like a normal sequence: (seq_len, heads, dim)
        K_seq = K_seq.view(-1, num_heads, head_size)[:seq_len]
        V_seq = V_seq.view(-1, num_heads, head_size)[:seq_len]
        
        # 3. Standard Attention Math for this specific sequence
        q_i = query[i].unsqueeze(0) # (1, heads, dim)
        
        # Transpose for dot product: Q @ K^T
        scores = torch.einsum('qhd,khd->qhk', q_i, K_seq) / (head_size ** 0.5)
        attn_weights = F.softmax(scores, dim=-1)
        
        # Weighted sum of values: Weights @ V
        out[i] = torch.einsum('qhk,khd->qhd', attn_weights, V_seq).squeeze(0)

    return out

Note: The real vLLM implementation does not actually gather and copy the memory into a contiguous K_seq tensor before computing attention—doing so would trigger the exact memory bottleneck we are trying to avoid! Instead, the custom kernel performs the dot-product computations chunk-by-chunk directly from the scattered physical blocks.

5. The Impact on Modern Architectures

PagedAttention fundamentally changed LLM serving:

  • Massive Batch Sizes: By eliminating VRAM waste, a single GPU can hold the KV cache for significantly more concurrent requests.
  • Continuous Batching: Because memory is dynamically assigned block-by-block, engines can safely inject new requests into the GPU at the exact millisecond an old request finishes, without waiting for the whole batch to complete.
  • Agentic State Sharing: If multiple agents share the same long system prompt, PagedAttention allows them to physically share the same memory blocks for that prompt, only diverging when they generate unique text.

6. The Drawbacks (The Engineering Reality)

While PagedAttention is the gold standard for throughput, it introduces specific trade-offs:

  • Loss of Coalesced Memory Access: GPUs are fastest when sweeping through unbroken, contiguous arrays. By forcing the GPU to jump around VRAM fetching scattered blocks, we take a slight penalty to pure computational speed compared to static pre-allocation.
  • Internal Block Fragmentation: We still waste a tiny bit of memory. If your block size is 16 tokens and a sequence stops at 17 tokens, you must allocate two full blocks. Those remaining 15 slots in the second block are wasted.
  • CPU Bookkeeping Overhead: Pre-allocation requires one memory operation at the start of a sequence. PagedAttention requires the CPU to actively monitor block capacity, hunt for free VRAM blocks, update the Block Table, and sync with the GPU at every generation step.
  • Kernel Complexity: You cannot just call torch.compile on PagedAttention. It requires maintaining highly complex, low-level custom kernels that must be re-written for different hardware architectures (NVIDIA, AMD, Apple Silicon).

Conclusion

PagedAttention proves that scaling machine learning systems isn’t just about training larger parameter models; it is about relentlessly optimizing the MLOps infrastructure that serves them. By breaking the contiguous memory rule, we traded a small amount of engineering complexity and CPU overhead for a massive leap in concurrent serving capacity.
